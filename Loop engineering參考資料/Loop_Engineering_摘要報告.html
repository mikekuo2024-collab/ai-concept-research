概念定義、起源背景與演進脈絡

從「打字提示」到「設計系統」——Loop Engineering 是 2026 年最重要的 AI 工程典範轉移

🔑核心定義

LOOP ENGINEERING — 一句話定義

Loop engineering 是「取代自己成為提示 AI agent 的人」——你不再手動輸入下一個指令，而是設計一套系統，由這套系統決定何時提示、提示什麼、以及結果是否可被接受。 這是一個遞迴式目標：定義一次目的，AI 持續迭代直到達成。

Addy Osmani 的定義（2026-06-07）

「Loop engineering 是取代自己成為提示 agent 的人，轉而設計做這件事的系統。A loop is a recursive goal — you define a purpose and the AI iterates until complete.」

addyosmani.com

Kilo AI 的定義（2026-06-10）

「Loop engineering 是設計、運作、改進反饋迴路的實踐，讓 AI coding agents 能夠計劃工作、修改代碼、觀察結果、修正方法，直到軟體任務完成。」

kilo.ai

2023
Autocomplete 時代 — AI 補全單行或函數，工程師握著工具、控制每一步，AI 只是更聰明的 Tab 鍵。
2024

2025
Multi-Agent 時代 — 並行運行多個 agent、Ralph Wiggum Loop 出現（Geoffrey Huntley 提出）。開始有人用 while 迴圈讓 agent 自動跑，每次都是全新 context。
2026 年 6 月

🏗️三層工程堆疊：從 Prompt 到 Loop

層次 優化對象 工作單元 核心問題
Prompt Engineering 如何措辭單一指令 你親手輸入的一個回合 怎麼說才能得到好答案？
Context Engineering 放進 context window 的內容 圍繞一個答案的完整條件 放什麼資訊讓答案更準確？
Loop Engineering 自動決定何時提示、提示什麼、結果是否可接受的系統 跨越多個回合的自動運行循環 怎麼設計系統讓它自己提示？

重要：Loop engineering 並未取代 Prompt engineering——loop 本身仍由 prompts 組成。它是更高一層的抽象，控制整個系統何時運行、何時停止、如何驗證。

🔄Ralph 技術：Loop Engineering 的原型

什麼是 Ralph Loop？

Geoffrey Huntley 在 2026 年初提出，以 Simpsons 角色 Ralph Wiggum 命名。核心思想：不追求一次完美，而是用最簡單的迴圈反覆執行，每次迭代都是全新的 agent 和乾淨的 context。

# The Ralph Loop — 原始形式
while ! grep -q "ALL TASKS DONE" STATUS.md; do
  claude -p "Read PLAN.md and STATUS.md.
             Pick the next unchecked task.
             Implement it, run tests, commit,
             then update STATUS.md. Stop."
done

# PLAN.md 和 STATUS.md 是唯一的持久記憶
# Agent 每次忘記一切；檔案記住一切

為什麼有效？

  • 每次重置避免 context 污染
  • 外部 state file 是唯一持久記憶
  • 任務粒度小、可驗證、可 commit
  • 極簡設計，任何人都能維護

Loop Engineering 的進化

  • while 迴圈 → 排程 Automation
  • Context 重置 → Worktree 隔離
  • 手動條件 → /goal 指令
  • Bash 腳本 → 內建產品功能

核心架構、操作流程與工具實作

⚙️核心循環：五步驟 Plan-Act-Verify

每一個 Loop 執行的基本單元，來源：Kilo AI（2026-06-10）

Intent — 意圖定義
開發者或排程系統定義目標結果。關鍵：目標必須具體、可驗證，例如「所有 test/auth 的測試通過且 lint 無錯誤」而非「讓 checkout 更好」。
Context — 上下文收集
Agent 收集相關代碼、文件、錯誤日誌、慣例規範。技巧：過多 context 反而有害，應聚焦與任務直接相關的資訊，並在重要觀察後重新收集。
Action — 執行行動
Agent 編輯文件、運行命令、呼叫工具（MCP connectors）。原則：優先做最小的可逆改動，小 diff 易於驗證和修復，大型重寫讓失敗難以追蹤。
Observation — 結果觀察
系統捕獲測試結果、編譯錯誤、runtime 輸出、lint 警告、瀏覽器截圖或 code review 評論。一個測試失敗不只是錯誤，是新的 context。
Adjustment — 計劃調整
Agent 根據觀察更新計劃並重複循環，直到停止條件達成或遇到需要人工判斷的阻塞點。Loop 的威力來自這個閉環，而非任何單一步驟。

🧱六大建構元素詳解

⏰
① Automations — 自動化（Loop 的心跳）
按時程觸發，自動執行探索與分類，無需人工開始每個循環。
/loop 設定排程；/goal 運行直到條件達成；Codex 的 Automations tab 支援 cron 排程與 Triage inbox。
🌿
② Worktrees — 工作樹（並行隔離）
讓多個 agent 並行工作時不互相覆蓋檔案，基於 git worktree 機制。Claude Code 支援 --worktree 旗標和 isolation: worktree 設定；Codex 內建每 thread 獨立 worktree。
📖
③ Skills — 技能（知識持久化）
將專案慣例、建構步驟、禁忌寫入 SKILL.md，讓 agent 不必每次從頭猜測。技能是知識的「寫下一次、永遠可用」——解決每次對話 agent 從零開始的問題。
🔌
④ Plugins & Connectors（MCP）
基於 Model Context Protocol 讓 agent 接入真實工具：issue tracker、資料庫、Slack、CI 系統。讓 agent 從「說做了什麼」進化到「真的做了什麼」——自動開 PR、更新 ticket、發送通知。
🤖
⑤ Sub-agents — 子代理（maker vs checker）
最重要的結構性設計：寫代碼的 agent 和審查代碼的 agent 必須分離。寫作者批改自己的作業太寬鬆。Checker 使用不同指令、有時用不同模型，在 loop 無人監看時是唯一的品質守門員。
🧠
⑥ Memory — 記憶（Loop 的脊椎）
markdown 檔、Linear board 或 GitHub issue——任何存在於對話之外的狀態。模型忘記所有；檔案記住一切。 明天早上的 loop 從今天停下的地方接著跑。

⚖️Claude Code vs OpenAI Codex 建構元素對照

兩個工具的設計形狀幾乎相同，選哪個工具不重要，重要的是設計同樣形狀的 loop。

🔁五種實用 Loop 模式（依任務類型）

🧪 Test-Driven Agent Loop

從失敗測試開始，agent 修改實作直到測試通過。適合 bug 修復、解析器行為、資料轉換邏輯。測試是唯一的 pass/fail 信號。

🔨 Compiler-Driven Loop

類型系統錯誤作為修復清單。適合型別遷移、依賴升級、重構。編譯器不能證明產品正確，但能快速抓住結構錯誤。

👁️ Review-Driven Loop

人工 code review 評論作為觀察來源。Agent 分類評論（bug、風格、產品判斷），對可執行的評論做修改，並驗證結果。

🐛 Runtime Debugging Loop

使用 log、stack trace、HTTP response 縮小問題。每次迭代：提出假設 → 最小修改或加入追蹤 → 觀察結果 → 更新假設。

🎨 Product Iteration Loop

UI/UX 工作的迭代：agent 實作版本 → 對照設計約束 → 調整排版 → 驗證響應式行為。觀察來源可能是瀏覽器截圖、無障礙檢查、語義 HTML 驗證。

🛑如何寫出 Loop 信任的停止條件

停止條件是「合約」，不是「願望」。弱停止條件讓 loop 自行決定何時算完成——通常太快。

欄位 ❌ 弱版本 ✅ 可驗證版本
End State 「提升測試覆蓋率」 「src/billing 的覆蓋率達到或超過 90%」
Evidence 「看起來做好了」 「npm test 退出碼 0 且覆蓋率報告確認數字」
Constraints （未說明） 「不得觸碰 public API 或刪除現有測試」
Budget （無限制） 「25 個回合或 $5 成本，以先到者為準」
# Claude Code — 可信任的 /goal 範例
/goal "All tests in test/auth pass and lint is clean. Do not modify public API signatures. Stop after 25 turns or $5, whichever comes first."

🗺️端到端 Loop 範例：一個早晨的自動分類循環

  1. 每日早上 9 點，排程 Automation 觸發
  2. 呼叫 $triage-ci Skill，讀取昨日 CI 失敗、open issues、近期 commits
  3. 寫入發現到 Memory 檔案（TODO.md 或 Linear board）
  4. 對每個值得處理的發現，開一個獨立 Worktree
  5. Maker sub-agent 在 worktree 起草修復
  6. Checker sub-agent 對照 Skills 和測試審查草稿
  7. Connector 自動開 PR 並更新 ticket
  8. Loop 無法處理的事項進入 Triage inbox 等待人工處理
TODO.md — Loop 的記憶
## Open
- [ ] flaky test in test/auth/login.spec.ts
      (CI run #4821)

## Done
- [x] bump axios to patched version
      (PR #312, merged 2026-06-20)
你設計了它一次。你沒有手動提示任何一個步驟。這就是 Steinberger 說「設計 loop 取代手動提示」的具體實現。

⚠️四大核心風險剖析

Loop 改變工作，不刪除你。以下三個問題隨著 loop 越來越好而變得更嚴重，不是更容易。

① 驗證仍是你的責任

無人監看的 loop 也是靜悄悄地犯錯的 loop。即使有 checker sub-agent，「完成」也是一個宣稱，不是證明。你的工作是 ship 你確認可用的代碼。

② Comprehension Debt（理解債）

代碼生產速度超過你理解的速度，擁有越來越多你不懂的 codebase。Loop 讓這個缺口增長更快，除非你主動閱讀 loop 生產的內容。

③ Cognitive Surrender（認知投降）

Loop 自動跑時，很容易停止有意見並接受輸出。設計 loop 是加速器（有判斷力）或風險因子（逃避思考）——行為相同，結果相反。你決定是哪一個。

④ Thrashing / 失敗模式

常見問題：Thrashing（反覆修改不收斂）、Overfitting to tests（測試過但需求未達）、Context drift（過時假設繼續工作）、Unsafe autonomy（執行破壞性操作）。

Addy Osmani 的警告：「兩個人可以建立完全相同的 loop，得到完全相反的結果。一個人用它更快完成自己深刻理解的工作；另一個人用它逃避理解工作。Loop 不知道差別在哪。你知道。」

✅五項最佳實踐

  • 從狹義任務開始 — 「修復 test/auth/login.spec.ts 中的 flaky test」優於「改進認證流程」。狹義任務讓 agent 清楚哪些檔案重要、哪些驗證輸出相關。
  • 告訴 agent 如何驗證工作 — 如果你知道正確的命令，寫進去。驗證指令將模糊的「進展」轉化為可測量的「完成」。沒有驗證的 loop 在代碼生成後就停止而不檢查結果。
  • 優先使用現有模式 — 好的 loop 尊重 codebase。Agent 應該檢查附近的實作、重用共用元件、遵循現有命名，除非任務明確需要新抽象。
  • 保留人工判斷主控權 — Agent 可以快速執行 loop，但人類仍然擁有產品意圖、風險承受度、架構決策和最終審查。設計 loop 的人決定在哪些節點讓人工介入。
  • 捕獲可重用的 Loop 模式 — 當一個工作流程運作良好，將它記錄為 Skill 或可重複的模式。標準化 bug 修復、依賴升級、PR 評論處理的 loop，降低每次重新發明的成本。
