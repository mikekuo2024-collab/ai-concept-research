概念定義與起源背景

從人類「邊想邊做」的認知模式，到 LLM 推理與行動的融合框架

核心定義
ReAct 是一個讓大型語言模型（LLM）在解決任務時，交錯生成「推理軌跡（Reasoning Traces）」與「任務行動（Task-specific Actions）」的框架。推理幫助模型制定、追蹤和更新行動計畫；行動使模型能與外部知識庫或環境互動，獲取額外資訊以支撐推理。
— 來源 [1] react-lm.github.io / [3] arXiv 2210.03629，Yao et al. 2022
觸發此概念的關鍵問題

2022 年前，LLM 推理（Chain-of-Thought）與行動（WebGPT、SayCan、ACT-1）被分開研究。CoT 擅長邏輯推理但缺乏事實接地（導致幻覺），而純行動代理則缺乏規劃能力。


「ReAct 的核心問：如果推理與行動兩種能力結合會如何？」


人類啟發：人類在執行新任務時（如用新廚具做菜），自然地結合語言推理（「這個旋鈕應該是調溫度的…」）和實際行動（轉動旋鈕、觀察結果）。ReAct 在 LLM 中模仿這個過程。

2022.11

2025-2026
多代理系統（Multi-Agent Systems）成為主流——ReAct 作為單個代理的核心迴圈，在更大規模的協作框架中持續扮演基礎角色

核心架構與操作流程

Thought → Action → Observation 迴圈的設計原理、元件拆解與程式碼實作

ReAct 的四大核心元件
💭
Thought（思維）
自然語言推理步驟。分析情境、規劃行動、處理例外，形成可追蹤的推理軌跡。
⚡
Action（行動）
具體操作命令，如 Search[關鍵詞]、Lookup[字詞]、Calculate[算式]、Finish[答案]。
👁
Observation（觀察）
外部環境的回應結果（搜尋結果、API 回傳）。作為下一輪 Thought 的事實輸入。
✅
Finish（完成）
特殊行動類型。當模型認為已收集足夠資訊時，輸出 Finish[最終答案] 終止迴圈。
標準操作流程（HotpotQA 範例）
1

接收問題

使用者輸入問題，如「科羅拉多河流域的最高峰海拔是多少？」

2

Thought 1：分析問題，制定搜尋計畫

「我需要搜尋科羅拉多河流域的資訊，找出哪些山峰位於該流域。」

3

Action 1：呼叫工具

Search[科羅拉多河流域] → 呼叫搜尋引擎

4

Observation 1：接收環境回應

「科羅拉多河流域涵蓋落磯山脈，Mount Elbert 海拔 4399 公尺…」

5

Thought 2：基於新資訊更新推理

「看到 Mount Elbert，但需確認是否為流域內最高峰，再搜尋確認。」

6

（重複 3–5）直到資訊充足

每次 Action 帶來新的 Observation，驅動下一輪 Thought，直到模型認為答案已確定。

7

Finish：輸出最終答案

Action N: Finish[Mount Elbert，海拔 4,399 公尺，為科羅拉多河流域最高峰。]

ReAct Prompt 格式（原論文範例）
# ReAct Prompt 格式（HotpotQA） Question: Aside from the Apple Remote, what other devices can control the program Apple Remote was originally designed to interact with? Thought 1: I need to search Apple Remote and find the program it was originally designed to interact with. Action 1: Search[Apple Remote] Observation 1: The Apple Remote is a remote control device... originally designed to control Front Row media center program... Thought 2: Front Row was originally the program. I need to search what other devices can control Front Row. Action 2: Search[Front Row (software)] Observation 2: ...can be controlled using the Apple Remote or the keyboard function keys. Thought 3: So besides the Apple Remote, the keyboard function keys can control Front Row. Action 3: Finish[keyboard function keys]
LangGraph 實作程式碼（2025 推薦）
# 安裝：pip install langgraph langchain-openai from langgraph.prebuilt import create_react_agent from langchain_openai import ChatOpenAI from langchain_core.tools import tool # 定義工具：docstring 是代理決定何時呼叫的依據 @tool def search(query: str) -> str: """搜尋網路取得最新資訊。query 為搜尋關鍵詞。""" # 實作搜尋邏輯... return f"搜尋結果：{query} 的相關資訊" @tool def calculator(expression: str) -> str: """計算數學表達式。expression 為有效的 Python 數學算式。""" return str(eval(expression)) # 建立模型與代理 model = ChatOpenAI(model="gpt-4o", temperature=0) tools = [search, calculator] # create_react_agent 封裝完整 Thought→Action→Obs 迴圈 agent = create_react_agent(model, tools) # 執行（自動串流思維軌跡） result = agent.invoke({ "messages": [("user", "台積電 2025 Q4 營收是多少？")] }) print(result["messages"][-1].content)

實戰指南、風險與完整資源

生產部署的注意事項、最佳實踐，以及本報告的完整參考來源

ReAct 採用路徑建議
1

概念驗證（PoC）

使用 LangChain 或 promptingguide.ai 範例，手動撰寫 2–3 個 ReAct few-shot 軌跡，測試 OpenAI / Claude API。

2

工具整合

定義 2–5 個業務相關工具（搜尋、資料庫查詢、計算），確保每個工具有清晰的 docstring，測試工具回傳格式。

3

升級到 LangGraph

用 create_react_agent 重構，加入狀態管理、max_iterations、串流輸出，準備生產部署。

4

評估與優化

收集失敗軌跡，分析是否為非資訊性搜尋（最常見）或重複迴圈，調整工具描述或考慮改用 Reflexion / ReWOO。

⚠️ 風險與注意事項
非資訊性搜尋（最高風險，佔 23% 錯誤）

搜尋結果不具信息性時，模型推理難以恢復。應設計 fallback 行動，並在工具中加入「無結果」的明確回傳格式。

重複迴圈陷阱

模型可能重複生成相同的 Thought 和 Action。必須設定 max_iterations（建議 10-20），避免無限 token 消耗。

結構性靈活度限制

嚴格的 T-A-O 格式有時會限制推理的靈活性，比 CoT 在某些數學推理任務上更差。對純推理任務考慮 CoT + ReAct 混合。

Token 成本隨迴圈線性增長

每次迴圈需重新發送完整上下文（包含所有歷史 T-A-O），長任務 token 成本高。預算敏感場景考慮 ReWOO。

✅ 最佳實踐清單
✓ 工具 docstring 要清晰：代理依賴 docstring 決定何時呼叫工具，模糊描述直接影響決策品質。
✓ 設定 max_iterations：生產環境必須設上限（建議 10-20），防止無限迴圈和意外高成本。
✓ temperature=0：確保代理行為一致可重現，避免非確定性輸出導致的難以調試問題。
✓ 複雜任務混合 CoT + ReAct：兩者結合的效果優於任一單獨使用，論文中確認為最佳配置。
✓ 加入可觀測性工具：LangSmith / Arize 追蹤完整軌跡，快速定位失敗的推理步驟。
✓ 高風險決策加入 Human-in-the-Loop：ReAct 的可解釋軌跡讓人類介入變得簡單——只需修改少數 Thought 就能糾正。
✓ 選擇正確架構：可預測批次任務用 ReWOO，高品質輸出用 Reflexion，一般探索任務才用 ReAct。
✓ 使用 LangGraph 而非舊版 AgentExecutor：2024 年後的生產代理應使用 LangGraph，支援狀態持久化和時間旅行除錯。
