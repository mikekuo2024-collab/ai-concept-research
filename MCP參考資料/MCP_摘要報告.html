概念定義與起源背景

MCP 是什麼？為什麼誕生？從 Anthropic 的官方定義到業界共識，追溯 Model Context Protocol 的起源脈絡。

核心定義
"MCP is an open standard that enables developers to build secure, two-way connections between their data sources and AI-powered tools. It provides a universal, open standard for connecting AI systems with data sources, replacing fragmented integrations with a single protocol."
— Anthropic 官方發布（2024年11月25日）

誕生背景：N×M 整合問題

在 MCP 出現之前，每個 AI 應用需要針對每個外部工具編寫自定義整合程式碼。若有 N 個工具（GitHub、Slack、資料庫⋯）和 M 個 AI 模型（Claude、ChatGPT、Gemini⋯），就需要 N × M 個各自獨立的適配器。

  • 工程成本爆炸性增長
  • 每次工具或模型更新，所有整合都需重寫
  • AI 模型被困在資訊孤島，無法取得即時資料

MCP 的解法：統一標準 → 只需 N + M 個實作，大幅降低整合成本。

演進歷史時間軸

2024年11月25日
Anthropic 正式開源 MCP，發布規格、SDK 與預建伺服器。由 David Soria Parra 和 Justin Spahr-Summers 主導開發。
2024年11月 – 2025年初
早期採用者跟進：Block、Apollo 整合 MCP；Zed、Replit、Codeium、Sourcegraph 宣布合作。Claude Desktop 率先支援本地 MCP Server。
2025年3月27日
OpenAI 正式宣布 Agents SDK 支援 MCP，並整合至 ChatGPT 桌面應用。MCP 升格為跨廠商行業標準。
2025年4月
安全研究員發布 MCP 安全評估報告，揭露命令注入、SSRF、提示詞注入等漏洞。社群開始強化安全最佳實踐。
2025年中
Google DeepMind 正式採用 MCP，Google Cloud（Vertex AI）整合 MCP 支援。三大 AI 巨頭均採用，MCP 成為事實標準。
2025年6月（最新）
協定版本 2025-06-18 發布，引入 Tasks（實驗性）、Elicitation 等新原語，進一步強化多步驟工作流支援。

核心架構與操作流程

深入 MCP 的技術架構：三層角色、三大原語、兩種傳輸機制，以及完整的 JSON-RPC 互動流程。

核心三層架構

🖥️

MCP Host（宿主）

用戶使用的 AI 應用程式，如 Claude Desktop、VS Code、Cursor。負責協調管理多個 MCP Client，包含 AI 協調邏輯，是整個系統的入口點。

🔌

MCP Client（客戶端）

存在於 Host 中的組件，與單個 MCP Server 維持 1:1 的專屬連接。負責訊息格式轉換、session 管理、錯誤處理和響應解析。

⚙️

MCP Server（伺服器）

提供工具、資源、提示的外部服務程式。可本地執行（如 Filesystem Server）或遠端部署（如 Sentry MCP Server）。暴露三種核心原語。

三大核心原語（Primitives）

🛠️

Tools（工具）

AI 可呼叫的可執行函式。每個 Tool 有唯一名稱、描述和 JSON Schema 輸入定義。可執行有副作用的操作：API 呼叫、資料庫查詢、計算等。

呼叫方法：tools/list → tools/call

📚

Resources（資源）

提供情境資訊給 AI 的唯讀資料來源。可以是檔案內容、資料庫記錄、API 響應等。資料由應用定義和控制，AI 不能透過資源修改系統。

呼叫方法：resources/list → resources/read

📝

Prompts（提示）

幫助結構化 LLM 互動的可重用模板。可包含系統提示、few-shot 範例，由 Server 暴露給 Client 使用，讓 AI 與特定工具的互動更有效率。

呼叫方法：prompts/list → prompts/get

完整互動流程（4 個步驟）

1

初始化（Lifecycle Management）

Client 發送 initialize 請求，與 Server 協商支援的協定版本和能力（capabilities）。雙方確認後，Client 發送 notifications/initialized 通知連接就緒。

2

工具探索（Tool Discovery）

Client 發送 tools/list 請求，Server 回傳可用工具清單（每個工具含名稱、描述、inputSchema）。AI 應用將所有連接 Server 的工具整合為統一工具庫。

3

工具執行（Tool Execution）

LLM 在對話中決定使用工具時，AI 應用攔截工具呼叫，路由到對應 MCP Server，執行 tools/call，將結果整合回對話。AI 獲得即時外部資料或執行操作。

4

即時通知（Notifications）

Server 可主動發送 JSON-RPC 通知（無 id，不需回應），告知 Client 工具清單或資源發生變更。Client 收到後自動重新拉取最新清單，保持資訊同步。

兩種傳輸機制對比

JSON-RPC 程式碼範例

初始化請求與回應（協議版本 2025-06-18）：

// Client → Server：初始化請求
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-06-18",
    "capabilities": { "elicitation": {} },
    "clientInfo": { "name": "my-app", "version": "1.0.0" }
  }
}

// Server → Client：初始化回應
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-06-18",
    "capabilities": {
      "tools": { "listChanged": true },
      "resources": {}
    },
    "serverInfo": { "name": "my-server", "version": "1.0.0" }
  }
}

// Python SDK 快速建立 MCP Server
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("My Weather Server")

@mcp.tool()
def get_weather(location: str, units: str = "metric") -> str:
    """Get current weather for a location"""
    # ... 實際 API 呼叫
    return f"Weather in {location}: 25°C, Sunny"

if __name__ == "__main__":
    mcp.run()  # 以 Stdio 模式執行

實戰指南、風險與完整資源

從入門到企業部署：MCP 的採用路徑、安全風險、最佳實踐，以及所有研究來源的完整索引。

MCP 採用成熟度路徑

🌱

初學者（0–2週）

安裝 Claude Desktop，連接官方預建 MCP Server（Filesystem、GitHub、Slack）。熟悉三大原語概念，嘗試讓 Claude 讀取本地檔案或查詢 GitHub。

🔧

開發者（1–4週）

使用 Python FastMCP 或 TypeScript SDK 構建第一個自定義 MCP Server，實作 1–3 個工具，連接到個人或公司的特定服務。

🏗️

中級（1–3個月）

部署遠端 MCP Server（Streamable HTTP），整合 OAuth 認證，構建多工具協作工作流，整合 LangChain 或 LangGraph 框架。

🏢

企業級（3個月+）

建立企業 MCP Server 市場，集中管理權限和認證，整合到現有 DevOps 流程，部署安全審計機制，連接 ERP/CRM 等核心系統。

安全風險警示

⚠️ 命令注入漏洞（43% 實作存在）

攻擊者可透過惡意工具輸入執行任意系統命令。解決方案：對所有工具輸入進行嚴格的白名單驗證，避免直接將用戶輸入傳入系統呼叫。

⚠️ SSRF（伺服器端請求偽造，30% 實作存在）

攻擊者可利用 MCP Server 探測內部網路或存取受保護資源。解決方案：限制 MCP Server 可存取的 URL 白名單，封鎖內網 IP 範圍。

⚠️ 任意檔案存取（22% 實作存在）

MCP Server 可能洩露系統設定檔、私鑰等敏感資料。解決方案：明確限制 MCP Server 可存取的目錄路徑，使用沙箱環境。

⚠️ 提示詞注入 + 工具投毒

惡意 MCP Server 可透過已連接的其他工具進行資料外洩，或操控 AI 執行未授權操作。解決方案：只連接可信任的 MCP Server，定期審查連接的 Server 清單。

最佳實踐清單

✅從簡單開始：先用官方預建 MCP Server（Filesystem、GitHub）熟悉概念，再開發自定義 Server
✅嚴格輸入驗證：對所有工具輸入使用 JSON Schema 驗證，拒絕不符合格式的請求
✅最小權限原則：MCP Server 只申請完成任務所需的最小系統權限
✅只連接可信 Server：維護已審核的 MCP Server 白名單，定期複審清單
✅使用 OAuth 認證：遠端 MCP Server 必須實作 OAuth 2.0，避免使用長效 API Key
✅早期投資可觀測性：記錄所有工具呼叫、輸入輸出、執行時間，便於除錯和安全審計
✅MCP 不取代框架：MCP 是整合層，複雜 Agent 邏輯仍需搭配 LangChain/LangGraph 使用
✅企業資料需額外保護：敏感企業資料須使用具企業級加密和合規認證的 MCP 解決方案
