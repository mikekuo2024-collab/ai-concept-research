第一頁 · 概念定義與起源

Vibe Coding：順應感覺的 AI 程式設計

從一則病毒式推文，到 Collins 年度詞彙，探索改變軟體開發文化的新範式誕生歷程。

🔍 核心定義

"There's a new kind of coding I call 'vibe coding', where you fully give in to the vibes, embrace exponentials, and forget that the code even exists." — Andrej Karpathy，OpenAI 共同創辦人，2025 年 2 月 X (Twitter) 貼文

Vibe Coding 是一種 AI 輔助的軟體開發實踐，開發者以自然語言描述需求，由大型語言模型（LLM）自動生成程式碼。其核心特徵是「接受 AI 輸出而不徹底審查」——開發者扮演的角色從「程式碼作者」轉型為「需求描述者」與「結果引導者」。

"我就是看到東西、說出東西、執行東西、複製貼上東西，大多數情況都能運作。" — Andrej Karpathy 描述其 Vibe Coding 工作流程

⚖️ Vibe Coding 的兩種應用模式

🌊
純粹 Vibe Coding
完全信任 AI 輸出，不手動審查程式碼。Karpathy 形容為「忘記程式碼的存在」。適合週末拋棄式專案、快速原型驗證，不適合生產環境或需要安全性的系統。
🛡️
負責任 AI 輔助開發
AI 作為協同工作者（Co-pilot），人類開發者負責引導、審查、測試並對所有程式碼負完全責任。這是企業和專業環境中推薦的實踐方式。

Andrej Karpathy 在 X 發出「淋浴後的隨想推文」，創造「Vibe Coding」術語。貼文超過 450 萬次瀏覽，引發全球熱議。
2025 年 3 月
Merriam-Webster 將「Vibe Coding」列為「俚語與潮流」用語，語義從科技社群向大眾文化擴散。
2025 年 4 月
MIT Technology Review 深度分析 Vibe Coding 的邊界條件與安全風險，主流媒體廣泛跟進報導。
2025 年末
Collins 英語詞典將 Vibe Coding 評選為「2025 年度詞彙」，確立其文化地位。
2026 年
Vibe Coding 成為主流開發方式。全球 92% 美國開發者採用，60% 新程式碼由 AI 生成。Karpathy 改稱「Agentic Engineering」強調人仍掌控主導。

🔄 Vibe Coding vs 傳統程式設計

特性傳統程式設計Vibe Coding
程式碼建立方式逐行手動撰寫AI 從自然語言生成
開發者角色架構師、實作者、除錯者提示者、引導者、測試者、精煉者
所需技術知識高（語言語法、框架、架構）低（理解功能需求即可）
主要輸入精確的程式碼自然語言提示與反饋
開發速度較慢、方法論式快速，特別適合原型設計
錯誤處理基於程式理解的手動除錯透過對話反饋精煉
學習曲線陡峭低門檻（任何人都能開始）
程式碼可維護性取決於開發者技術水準高度依賴 AI 輸出品質與人工審查

👥 誰適合 Vibe Coding？

🧑‍💻
有程式基礎的開發者
用於快速原型驗證、內部工具開發、自動化腳本生成。了解程式設計讓他們能識別 AI 的錯誤並快速修正。
🌱
完全沒有程式背景的人
MIT AI 研究員 Tobin South：「我定義 Vibe Coding 是『有個 AI 能執行但你不會實現的願景』。」適合低風險的個人專案和小型 App。

🧩 核心組成元素

🗣️
自然語言介面
用英文或中文描述需求，取代傳統的語法撰寫。「最熱門的程式語言是英文」——Karpathy。
🤖
大型語言模型 (LLM)
Claude、GPT-4、Gemini 等 AI 理解需求並生成程式碼。提供多輪對話式精煉能力。
🔄
迭代反饋迴圈
描述→生成→測試→反饋→精煉的循環。錯誤訊息直接貼回 AI，讓它自行修復。
🛠️
AI 開發環境 (IDE)
Cursor、Windsurf 等工具將 AI 深度整合到 VS Code，提供即時建議和多檔案生成。
🔌
Model Context Protocol
Anthropic 制定的開放協議，讓 AI 代理能連接外部工具、資料庫、API，擴展能力邊界。
🚀
Vibe Deploying
Vibe coding 的延伸：一鍵將應用部署到 Cloud Run 等生產環境，消除 DevOps 瓶頸。

⚙️ 主要操作流程

【程式碼層】緊密迭代迴圈

描述目標（自然語言提示）
例：「建立一個 Python 函數，讀取 CSV 檔案並回傳 email 欄位的所有值。」
AI 生成程式碼
LLM 解讀需求，生成完整的程式碼，包含函數、邏輯和基本錯誤處理。
執行並觀察
運行生成的程式碼，觀察輸出是否符合預期。若有錯誤，直接複製錯誤訊息。
提供反饋並精煉
例：「這樣可以，但請加上找不到檔案時的錯誤處理。」或直接貼上錯誤訊息讓 AI 修復。
重複直到完成
繼續描述-生成-測試-精煉的循環，直到程式碼達到預期功能。

【應用程式層】從構思到部署

構思
用一個高階提示描述整個應用程式的功能、UI 和需求。
生成
AI 生成完整的應用（UI、後端邏輯、檔案結構、資料庫 schema）。
迭代精煉
測試應用，用後續提示添加功能或修改設計。
測試與驗證
人工（或 AI 輔助）審查安全性、品質和正確性。
部署
一鍵部署到雲端平台（如 Cloud Run），讓應用程式上線。

第三頁 · 實戰指南與資源

負責任地 Vibe Coding：風險、最佳實踐與學習資源

掌握 Vibe Coding 的邊界條件、安全威脅防範，以及完整的學習資源清單，讓你高效且安全地實踐 AI 開發。

🎯 何時使用 vs 何時避免

✅ 適合的場景

  • 快速原型驗證（週末專案、概念驗證）
  • 個人工具和小型 App（不儲存敏感數據）
  • 內部工具開發和自動化腳本
  • 學習新框架或語言（有 AI 輔助學習）
  • 簡單的靜態網頁和展示頁面
  • 數據處理和分析腳本

⛔ 不適合的場景

  • ❌ 核心業務系統（金融交易、醫療記錄）
  • ❌ 大規模用戶數據的應用（需要嚴格安全審查）
  • ❌ 監管要求嚴格的行業（金融、醫療、法律）
  • ❌ 需要高可維護性的長期生產系統
  • ❌ 安全性要求高的公開 API

⚠️ 安全風險與注意事項

🔴 統計警告：45% 的 AI 生成程式碼含已知安全漏洞

研究表明，約 45% 的 AI 程式碼生成任務會引入至少一個已知安全缺陷。經過五次迭代後，程式碼可能比初始版本多出 37% 的嚴重漏洞。

🔴 SQL 注入與命令注入

AI 模型偏向用字串拼接方式構建資料庫查詢，導致經典 SQL 注入漏洞。

🔴 硬編碼憑證

AI 生成程式碼常將 API 密鑰、密碼直接寫入程式碼，造成安全洩漏風險。

🔴 不安全的依賴項

AI 自動引入開源套件，若不驗證，可能繼承有漏洞或惡意的第三方庫，擴大攻擊面。

🔴 心理距離弱化問責制

開發者對 AI 生成的程式碼感覺「不是自己寫的」，導致責任感降低，安全審查更容易被跳過。

✅ 最佳實踐清單

  • 在提示中明確要求安全考量（「確保防止 SQL 注入，使用參數化查詢」）
  • 所有 AI 生成程式碼必須由人類開發者審查，不接受未理解的程式碼
  • 整合 SAST（靜態應用安全測試）到 CI/CD 流程，自動掃描漏洞
  • 使用 SCA（軟體組成分析）驗證所有 AI 引入的第三方依賴項
  • 敏感資料永遠不要在提示中出現（API 密鑰、密碼、個人資料）
  • 建立 CLAUDE.md 或 GEMINI.md 等上下文檔案，給 AI 設定專案標準
  • 為每個 AI 生成的函數撰寫單元測試（可以再讓 AI 幫你寫測試）
  • 部署前進行手動安全滲透測試，特別是公開 API 和用戶輸入處理
  • 採用最小權限原則：AI 生成程式碼默認請求最小必要系統權限
  • 版本控制所有 AI 生成的程式碼，保持完整的修改歷史記錄
