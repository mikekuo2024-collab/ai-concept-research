Context Engineering 是如何誕生的？它的核心定義是什麼？與過去的 Prompt Engineering 有何本質差異？

核心定義（最廣被引用）
"Context Engineering 是設計與建構動態系統的學門，在正確的時機、以正確的格式，提供正確的資訊和工具，讓 LLM 能完成任務。"
— Philipp Schmid（Google DeepMind），2025-06-30

核心架構與操作流程

Context Engineering 的七大組成元素是什麼？四大支柱如何運作？實際的程式碼和工具比較。

Context 的九大組成元素

Context 不只是你傳送的 prompt，而是模型回應前所見到的一切（Philipp Schmid, LlamaIndex 整合版）

📋

System Prompt / Instructions

定義模型行為的初始指令、規則、人設，設定整個對話的場景。

💬

User Prompt / Input

使用者的即時任務、問題或指令，是觸發 LLM 生成的直接輸入。

🔄

State / Chat History（短期記憶）

當前對話中的所有訊息歷史，讓模型理解對話脈絡和前後文。

🧠

Long-Term Memory（長期記憶）

跨對話持久化的知識：使用者偏好、歷史摘要、已學習的事實。

🔍

Retrieved Information（RAG）

從外部文件、資料庫、API 動態檢索的即時相關知識。

🔧

Available Tools（工具定義）

所有可呼叫函數的定義和 schema，讓模型知道有哪些能力可用。

📤

Tool Responses（工具回應）

工具執行後的回傳結果，作為模型下一步推理的輸入資料。

📐

Structured Output（格式定義）

定義模型回應的格式（如 JSON schema），既控制輸出也提供結構化輸入。

🌐

Global State / Workflow Context

多步驟 Agent 中的全域變數、任務進度、先前步驟結果的暫存區。

四大核心操作支柱（Lance Martin, LangChain）

  1. Write（寫入）— 建立長期記憶

    動態將重要資訊儲存到 context window 之外，形成長期記憶。跨對話記住使用者偏好、任務狀態、學習到的事實。工具：向量資料庫、Mem0、LangGraph checkpointing。

  2. Read（選擇）— 按需注入 Context

    從記憶、工具庫、知識庫按需拉取相關資訊注入 context window。包含：從長期記憶召回、動態選擇工具、RAG 檢索。目標：最小可行 context。

  3. Compress（壓縮）— 控制 Token 用量

    當 context 過長時，僅保留執行任務所需的 tokens。技術：對話摘要、Context Pruning（移除過時資訊）、Context Offloading（卸載到外部工作區）。Claude Code 超過 95% 時自動 auto-compact。

  4. Isolate（隔離）— Multi-Agent 架構

    將任務拆分給 Sub-Agent 執行，Lead Agent 只接收結果而非完整執行過程。大幅節省 Lead Agent 的 context 用量，但需注意資訊共享的挑戰。

實戰指南、風險與完整資源

如何從 Prompt Engineering 升級到 Context Engineering？有哪些風險要避免？完整來源清單。

Context Engineering 採用成熟度路徑

Level 1

靜態 Prompt

只有 system prompt + user prompt，無動態 context

Level 2

RAG 整合

加入外部知識庫檢索，基礎 context 擴充

Level 3

記憶系統

短期 + 長期記憶，跨對話持久化使用者狀態

Level 4

動態工具選擇

按任務動態注入工具定義，而非預載所有工具

Level 5

完整 CE 系統

壓縮 + 快取 + 多代理隔離，生產級 token 優化

風險與注意事項

Lost in the Middle 問題

研究顯示 LLM 對 context 中間位置的資訊注意力最低。避免將關鍵資訊埋在過長 context 的中段；使用排序技術讓最重要的資訊出現在開頭或結尾。

Context 成本失控

生產級 Agent 可能進行數十至上百次工具呼叫。若每次工具輸出都原封不動塞回模型，token 費用將呈指數增長。務必實作壓縮管線和 Context Offloading。

Context 品質的上游問題

多數討論聚焦在如何管理 context，但忽略「context 來源治理」：企業 context 從哪裡來？誰在維護？過時、不完整、未治理的 context 會導致即使架構完美也無法正常運作。

Multi-Agent 資訊共享問題

隔離 context 到 Sub-Agent 雖然節省 token，但會導致子代理之間無法有效共享中間狀態和推理過程。需要在隔離效率和資訊共享之間取得平衡。

過度工程化（Over-Engineering）

目標是「最小可行 context」，而非「最大化 context」。塞入不相關的資訊反而會分散模型注意力、降低性能（Garbage In, Garbage Out）。先從簡單開始，按需求增加複雜度。

最佳實踐清單

  • 從「最小可行 context」原則出發，只提供完成任務所需的最少資訊
  • 設計 context 管線系統，而非只寫靜態 prompt 字串
  • 根據任務動態決定要包含哪些工具定義，避免工具定義膨脹
  • 為 context window 設定預算，監控每次 LLM 呼叫的 token 使用量
  • 實作對話摘要機制，超過閾值時自動壓縮歷史記錄
  • 使用結構化輸出（JSON Schema）確保工具回應格式一致、易於再利用
  • 建立長期記憶的更新和過期機制，防止過時資訊污染 context
  • 採用混合搜尋（Hybrid Search = 向量 + BM25）提升 RAG 檢索精準度
  • 利用 Prompt Caching 快取常用的 system prompt 和知識文件，降低延遲和成本
  • 在複雜任務中使用 Multi-Agent 隔離，但設計清晰的 Sub-Agent 回傳格式
  • 建立 context 品質監控：相關性、新鮮度、完整性、精簡性四指標
  • 思考「context 來源治理」：確保外部知識庫定期更新、有明確的負責人
